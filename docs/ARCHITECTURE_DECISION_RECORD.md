# 아키텍처 결정 기록 (ADR)

## ADR-001: JWT 토큰 기반 인증 선택

### ADR-001 상태

승인됨

### ADR-001 배경

세션 기반 인증 vs 토큰 기반 인증

### ADR-001 결정

JWT 토큰 기반 인증 채택

### ADR-001 이유

- 서버 확장성 (Stateless)
- 마이크로서비스 아키텍처와의 호환성
- 모바일 앱 지원 용이

### ADR-001 대안 고려

- OAuth 2.0: 복잡도 증가로 제외
- 세션 기반: 확장성 문제로 제외

## ADR-002: QueryDSL 도입

### ADR-002 상태

승인됨

### ADR-002 배경

동적 쿼리 작성 방법 선택: 문자열 기반 쿼리 vs 타입 안전 쿼리 빌더

### ADR-002 결정

QueryDSL 도입

### ADR-002 이유

- 타입 안전성: 컴파일 타임에 쿼리 오류 검출
- 컴파일 타임 에러 검출: IDE 자동완성 및 리팩토링 지원
- 복잡한 동적 쿼리 작성 용이: 조건부 쿼리 작성이 간결하고 가독성 높음
- 리팩토링 안전성: 엔티티 필드명 변경 시 자동으로 쿼리도 함께 변경

### ADR-002 대안 고려

- 문자열 기반 동적 쿼리: 타입 안전성 부족, 런타임 오류 가능성으로 제외
- Criteria API: 복잡한 문법과 가독성 저하로 제외

## ADR-003: JPA 선택

### ADR-003 상태

승인됨

### ADR-003 배경

데이터 접근 계층 기술 선택: MyBatis vs JPA

### ADR-003 결정

JPA (Java Persistence API) 채택

### ADR-003 이유

- **객체 지향 프로그래밍과의 자연스러운 통합**: 엔티티를 객체로 다루어 도메인 모델과 데이터베이스 간의 불일치 최소화
- **생산성 향상**: Spring Data JPA를 통한 CRUD 작업 자동화 및 반복 코드 감소
- **타입 안전성**: 컴파일 타임에 쿼리 오류 검출 가능
- **코드 가독성**: SQL 쿼리보다 선언적인 방식으로 비즈니스 로직에 집중 가능
- **유지보수성**: 엔티티 변경 시 자동으로 스키마 변경 반영 (JPA DDL 자동 생성)
- **Spring 생태계 통합**: Spring Boot와의 완벽한 통합 및 자동 설정 지원

### ADR-003 대안 고려

- **MyBatis**: SQL 매핑 파일 관리의 복잡성, 반복적인 CRUD 코드 작성 필요, 객체 관계 매핑의 수동 처리로 인해 제외
